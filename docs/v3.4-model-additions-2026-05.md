This document records the open-weights model additions made in Ragbot v3.4, the canonical source consulted for each model's facts, and the rationale behind the sizing-matrix verdicts. ## Rationale for sizing-matrix verdicts

The `comfortable / tight / Q4-only / won't fit` verdicts in `docs/sizing-matrix.md` derive from three principles:

1. **70-75% rule for unified-memory budget.** macOS, the runtime, other applications, and KV cache collectively claim ~25-30% of unified memory. A model whose weight footprint exceeds 70-75% of total unified memory is `tight` even if it technically loads. The exception is dedicated-inference Mac Studio configurations where the budget can stretch to 85-90% if the user runs no other apps.

2. **MoE quantization sensitivity.** Q4 on dense models is usually fine; Q4 on MoE models can produce routing degradation. For MoE entries (Scout, Maverick, Qwen3.6-35B-A3B, DeepSeek V3.2, Mistral Large 3, Mistral Small 4), the matrix prefers Q5 or Q8 where memory allows and explicitly notes the Q4-only tier as a quality trade-off, not just a memory choice.

3. **Memory bandwidth as the dominant tok/s factor.** The `tokens/sec ≈ bandwidth / model_size` heuristic guides the published per-Mac decode-speed expectations. MoE models scale by active params (not total weights), which is why a 109B-active-17B Scout decodes faster than a 31B-dense Gemma 4 31B on the same Mac despite the larger total footprint.

## Architectural decisions

### Why every local-inference entry has its own `local_inference` block

The pre-v3.4 schema used a flat structure. v3.4 introduces a nested `local_inference: { mlx_supported, ollama_tag, llama_cpp_quants }` block so the schema can express future MLX-specific or llama.cpp-specific fields without polluting the top-level. The `backend_notes` field is a sibling block for free-form performance notes (MLX tok/s, quantization caveats) that don't slot cleanly into structured fields.

### Why we kept the existing Gemma 4 entries unchanged at the ID level

Per the Ragbot CLAUDE.md rule, "Models in engines.yaml should ONLY move forward, NEVER backward." The existing Gemma 4 entries are pinned at `gemma4:e4b`, `gemma4:26b`, `gemma4:31b`. v3.4 adds metadata to these entries (the new `local_inference` and `backend_notes` blocks) without changing the model name or downgrading any configuration. Users with Ollama already configured for these models see no change at the runtime level.

### Why Llama 4 Scout's 10M context window is documented but not the recommended working configuration

The 10M context is the model's theoretical capability. At 10M tokens, the KV cache memory is ~390 GB even at FP16 — beyond any Apple Silicon configuration. Realistic workloads use 64K-1M of the available context window. The sizing matrix calls this out explicitly per profile.

### Why DeepSeek V3.2 has the largest `default_max_tokens` in the ollama provider (8192)

DeepSeek V3.2's reasoning workloads tend to produce longer outputs than chat workloads. The model's `max_output_tokens` is 65,536; `default_max_tokens` of 8,192 (vs. the standard 4,096 for other ollama models) leaves comfortable room for typical reasoning responses without users needing to override per-call.

## Verification

Schema-level tests for the v3.4 model additions live at [`tests/test_engines_yaml.py`](../tests/test_engines_yaml.py). The test suite covers:

- Top-level engines.yaml structure (parses, has required keys, provider names unique).
- Each new v3.4 model is present with the expected category, provider, and minimum context window.
- Each new v3.4 model declares the required `local_inference` fields, license, and `backend_notes.mlx`.
- Thinking-mode declarations use valid enum values.
- Existing Gemma 4 entries gained MLX-backend metadata without losing their original schema.
- Cross-cutting integrity (no duplicate names, ollama provider needs no API key, default models exist, output-token consistency).
- Loader integration tests skip gracefully when `synthesis_engine.config` is mid-refactor; the schema-level tests are the canonical contract.

Run from the repo root: `PYTHONPATH=src python3 -m pytest tests/test_engines_yaml.py`.

## Gaps and open questions

1. **MLX-native model availability is moving fast.** The `ollama_tag` field points at the standard tag; users with sufficient memory may prefer the MLX-native variants (`qwen3.6:27b-mlx-bf16`, `mlx-community/DeepSeek-V3.2-mlx-5bit`). A future schema update could add a `mlx_native_tag` sibling field. Not in v3.4.

2. **The sizing matrix assumes single-model loaded.** Multi-model warm-loading is discussed in the cross-cutting section but the per-hardware tables show single-model verdicts. A future revision could add a "two-model warm" verdict column for users running A/B comparison workflows.

3. **No public Mac Studio M5 Ultra benchmarks yet.** The M5 Ultra is expected mid-2026. When real benchmark numbers land, the sizing matrix's Mac Studio profiles will need a refresh. The current numbers are for the M3 Ultra; the entry-level Mac Studio for new buyers in mid-2026 will likely be M5 Ultra and run all listed models faster than the matrix currently shows.

4. **The Llama 4 Community License is not OSI-approved.** Mistral Small 4, Qwen3.6, and Mistral Large 3 are Apache 2.0. DeepSeek V3.2 uses the DeepSeek Model License (permissive but not Apache). Mistral Medium 3.5 is Mistral Research License (commercial use requires a paid license). Users with hard "must be OSI-approved" requirements should filter on the `license` field accordingly.
